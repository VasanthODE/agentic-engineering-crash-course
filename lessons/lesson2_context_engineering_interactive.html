Context engineering

The model is stateless. Everything it knows for a response comes from what's in the context window right now. This lesson is the craft of filling that window — the right information, in the right place, at the right time.

When an agent answers wrong, the first question is never "is the model dumb?" It's "what was in the context window?" Nine times out of ten the fix is context engineering — and the feedback loop is instant.
01 The highest-leverage block

System prompt design

A weak prompt describes ("be helpful"). A strong prompt constrains — defining exactly who the model is, how to shape output, and what it must never do. Three components, each load-bearing.

1 · Role 2 · Output format 3 · Prohibitions Weak vs strong
Prohibitions are the block beginners skip and experts obsess over. They encode legal, financial, and brand risk. The strongest form pairs each "never" with an escape route ("instead, direct to..."), which the model follows far more reliably.
02 Show, don't tell

Few-shot examples

Stop describing the task and start demonstrating it. Two or three input→output examples become a pattern the model instinctively completes — often more reliable than a page of careful instructions.

Describe it (instructions)
Show it (few-shot)
Few-shot is the cheaper cousin of fine-tuning. Both teach a behavior — fine-tuning bakes it into weights (slow, permanent); few-shot puts the demonstration in context (instant, free to change). Cover the edge cases, not just the easy middle.
03 Reasoning as scaffolding

Chain-of-Thought

The model thinks by writing — each token it generates becomes context for the next. Force an immediate answer and it must compress all reasoning into one leap. Let it write the steps and each becomes scaffolding the answer stands on.

Answer immediately
Think step by step
Same question, two prompting styles
A store had 120 items. It sold 25% on Monday, then 40% of what remained on Tuesday. How many are left?
▶ Run the model
Match the technique to the stakes. Reasoning tokens cost money, add latency, and fill the window (O(n²)). For a high-stakes refund decision, worth it. For a simple lookup, pure waste.
04 Data is not instructions

XML structuring

As your window fills with instructions, examples, retrieved docs, and user text, the model sees one undifferentiated stream. Delimiters draw clean boundaries — so a malicious line in a retrieved document stays data, not a command.

Flat text (no structure)
XML-delimited
A support agent answering from a retrieved review — which contains a hidden injection
Any content that didn't come from you — wrap it. Retrieved docs, user uploads, tool outputs, web content. Tag it, declare it data, and the model treats it as data. This is the #1 habit for RAG agents.
05 Room to be wrong on the way to right

Extended thinking

Question: refund on a damaged item delivered 35 days ago. Policy: 30-day returns, but damaged items get 7 days from a report. They reported damage on day 5. Approve?

Press run. Watch the model explore the policy conflict privately — including a self-correction — before producing a short, correct answer.
Reserve it for genuinely hard problems. Thinking tokens are billed like output and subject to O(n²). You set a thinking budget per task — a dial you tune to the stakes. It pairs with every other technique; it's the deepest layer, not a replacement.
The discipline: the model is a stateless predictor whose whole world is the context window you build. A system prompt sets role, format, and prohibitions · few-shot teaches by demonstration · Chain-of-Thought scaffolds the answer with reasoning · XML keeps data from being mistaken for instructions · extended thinking adds private room to self-correct.
