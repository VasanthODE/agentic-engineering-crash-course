01 Why MCP exists

The N×M integration problem

Every external system has its own API, auth, and format. If every app writes a custom integration to every tool, the connections explode multiplicatively. MCP collapses that to one standard connector.

"The USB-C of AI." Before USB-C, every device had its own connector and a drawer of adapters (N×M). One standard port fixed it. MCP is that, for connecting AI apps to tools and data — an open standard so the ecosystem converges on one connector.
02 What it is

Host · client · server, and three primitives

MCP is client-server. Your app is the host; it creates one client per connection; each client talks to one server that fronts a real system. And a server can expose three different things — not just tools.

Tools for the model, resources for the app, prompts for the user. Three primitives, three controllers. Most tutorials only mention tools — knowing resources and prompts, and who drives each, is the real signal you understand MCP.
03 How it moves

The connection lifecycle

Two phases: setup (handshake + discovery — the part unique to MCP) and runtime (the ReAct loop from Lesson 4, now driven by discovered tools). Discovery is what makes MCP dynamic rather than hardcoded.

▶ Next step Reset step 0
MCP changes how tools get INTO the loop, not how the loop runs. Discovery dynamically registers a server's tools into the model's context; everything after the model decides to act is the exact ReAct loop — route, call, result, echo tool_use_id, re-invoke.
04 The risk

Security & trust

The moment your agent dynamically trusts tools from external servers, you've handed partial control to whoever wrote them. You trust not just a server's code, but every word of text it sends your model.

Tool poisoning
Threats & defenses
A tool from an untrusted server. Its description looks innocent — until you reveal what's hidden. Recall from Lesson 4: the model reads the description as its decision signal.

The broader attack surface once your agent trusts external servers. Click each threat to see the attack and its defense:
Trust must be deliberate, scoped, and revocable. Connect only vetted servers; treat every byte a server sends (descriptions AND results) as untrusted; give each least-privilege credentials; put a human in the loop for anything irreversible. This is the on-ramp to Lesson 7's guardrails.
The arc: connecting agents to the world creates an N×M explosion, which MCP collapses to N+M · the standard is a host/client/server architecture exposing tools, resources, and prompts · the lifecycle is discovery (dynamic tool registration) then the familiar ReAct loop · and because trust is dynamic and external, security is first-class. MCP tools ARE Lesson 4's definitions, discovered at runtime — built entirely on the foundations beneath it.
