01 The distinction

Two questions, two tools

An agent is non-deterministic, multi-step, and calls external systems — so "it works" needs proof. Two complementary capabilities provide it. A concrete problem makes the split clear.

A problem arrives: users report the agent is giving wrong refund amounts. Two different tools, two different questions:
Observability"what happened in THIS run?"
Evaluation"is the agent good ACROSS runs?"
trace · run #8f2a · one user's failing request
eval suite · run on every change · refund test cases
pass rate
You need both. Observability is the microscope (one run, reactive, debug); evaluation is the dashboard (many runs, proactive, gate). Together: the difference between "I think it works" and "I know it works — and I'll know the moment it stops." These are the production agent's tracer.py and eval/suite.py.
02 The microscope

Tracing

A single agent run is a tree of nested steps. A trace is the whole run; a span is one unit of work (an LLM call, a tool call, a guardrail). The tree mirrors the ReAct loop.

The trace tree
How it's built · @trace
One agent run — a refund request that looped twice. Each bar is a span; click any to inspect it:
0ms500100015001950ms
A span doesn't appear by magic. The @trace decorator wraps a function so it records itself:
@trace # ← wraps the function as a span def calc_refund(order_id, reason): # your normal logic return compute(order_id, reason)
↓ produces ↓
span: calc_refund
trace_id8f2a… (shared by all spans in the run) parentagent.run (from the call stack) start → end1540ms → 1660ms (120ms) statusok inputs{order_id: 1234, reason: "damaged"} output$50.00
The decorator records the start, runs the function, records the end + status + result, and links the span to whatever @trace function called it — so nesting comes free from the call stack. Every span shares one trace_id, which reassembles scattered steps into one tree. Three things every span captures:
Timing
start, end, duration — where latency goes
Status
ok / error — where it failed
Attributes
inputs, outputs, tokens, cost, model
This is the Span / Trace / @trace model in your production agent's tracer.py — the same OpenTelemetry-style distributed tracing used to debug microservices. Agents need it even more: one run fans out into many non-deterministic steps a flat log can't untangle.
A trace answers three questions at a glance: where's the latency (longest bars), where's the cost (token-heavy LLM spans), and where did it fail (the error span). The tree structure — not a flat log — is why tracing is essential for multi-step agents.

04 Proactive quality

Evaluation

An eval suite is unit tests for your agent: golden cases with known answers, run on every change. The hard part isn't running them — it's grading a fuzzy, non-deterministic output.

Grading the fuzzy output
The suite & pass rate
Three graders — click each to see what it catches and misses:
A suite mixes graders, one per case's output type. The pass rate is your aggregate quality score:
pass rate
Match the grader to the output type — exact match for structured, tool-call for behavior, LLM-judge for fuzzy text. The suite is ALIVE: every production failure becomes a new golden case. Evaluation is how you manufacture the verifiable feedback (Lesson 4) that any domain needs.
05 Enforcement & the loop

The deploy gate

An eval suite is just a report until it has teeth. The deploy gate wires it into CI: a drop in pass rate blocks the deploy. Bad changes physically can't reach users — test-gated CI for agent quality.

The gate
The flywheel

The flywheel binds the whole lesson: Deploy → Observe → Catch a failure → Add a golden case → Improve + gate → Deploy. Observability (reactive) feeds evaluation (proactive) at the "add a case" hinge. Ship with confidence, not prayer.
The two halves of knowing it works. Observability is the microscope — traces, logs, metrics — answering "what happened?" reactively. Evaluation is the dashboard — golden cases, graders, pass rate, deploy gate — answering "is it good?" proactively. The flywheel binds them: observe → catch → add a case → improve → gate → deploy. This is the lesson that makes every other lesson trustworthy — and evaluation is how you manufacture the verifiable feedback that made coding the killer app (Lesson 4).
