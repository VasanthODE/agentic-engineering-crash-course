01 The heartbeat

The ReAct loop

An agent doesn't answer in one shot. It takes an action, observes the result, decides the next action from what it just saw, and loops — until the goal is reached. Reason → Act → Observe → repeat.

Step through a real agent trace. Watch each action get chosen from the previous observation — that feedback is what makes it an agent:

The agent decides when it's done. The loop isn't a fixed length — when the model stops calling tools and produces prose, the goal is reached. One question here took two tool calls, each chosen live. That feedback is the whole difference between a chatbot and an agent.
02 The production reality

Loop mechanics & the stop_reason trap

The loop is code you write, not something the model does. The model responds once and stops; your runtime inspects the response, runs tools, feeds results back, and calls again. The whole control flow hinges on one field: stop_reason.

stop_reason has more than two values — handle every one. The fragile loop assumes "tool or done." The robust loop branches on tool_use, end_turn, AND max_tokens. Also: every tool result must echo the exact tool_use_id from the request, or the API rejects the turn.
03 What the loop acts through

Tool definitions

When an agent calls the wrong tool, the bug is almost never your code — it's your descriptions. The model chooses tools by reasoning over the text you wrote. The description isn't documentation; it's the decision signal.

Write descriptions for the model, not for yourself. State what each tool does AND what it does not do — "does NOT access specific orders" is what lets the model rule out the wrong choice. Use enums for fixed values, examples in parameter descriptions, and mark required fields so the agent asks for what it's missing.
04 When one agent isn't enough

Multi-agent patterns

The skill isn't reaching for many agents — it's knowing when one won't do, then matching the topology to the task's structure. Independent subtasks parallelize; dependent ones sequence; high stakes get a critic.

Click each pattern to see its topology, data flow, and the task structure it fits:
Match the topology to the task, and prefer the simplest that works. Single agent → most of the time. Dependency chain → pipeline. Independent subtasks → parallel orchestrator. Costly errors → debate/critic. Every extra agent multiplies cost, latency, and failure modes.

Coding gives an agent a tight loop over a verifiable, observable, reversible environment. That's the ideal habitat for ReAct — and why it broke through first. When sizing up any agent problem, ask: can it verify success, see the full state, and undo mistakes?
